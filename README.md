# Trailing Period URL Encoding — Proof of Concept

This is a small PHP app that empirically tests browser behaviour around
percent-encoded trailing periods in URLs, built in the context of
[Wikimedia Phabricator task T423306](https://phabricator.wikimedia.org/T423306).

## The problem

Wikipedia article titles can end with a period — e.g.
[Mata v. Avianca, Inc.](https://en.wikipedia.org/wiki/Mata_v._Avianca,_Inc.)

When that URL is shared as plain text in a markdown-aware platform (Phabricator,
GitHub, Lemmy, etc.), the linkifier treats the trailing period as
sentence-ending punctuation and strips it, producing a broken link:

https://en.wikipedia.org/wiki/Mata_v._Avianca,_Inc.

## The proposed fix

MediaWiki's `wfUrlencode()` function (and its JavaScript counterpart
`mw.internalWikiUrlencode()`) should percent-encode a trailing period as
`%2E`, producing:

https://en.wikipedia.org/wiki/Mata_v._Avianca,_Inc%2E

Markdown linkifiers see no trailing period to strip, so the link survives.

## A question worth investigating

A commenter on T423306 wondered whether this fix would actually hold up in
practice: even if the server generates `%2E`, browsers decode it to `.` in
the address bar — so would a user copying and sharing the URL end up with a
bare period again anyway?

## What this POC shows

This app serves pages under `/wiki/<title>` with no redirects, and displays
exactly what `REQUEST_URI` the server received. It has three test links:

| Link | `href` attribute |
|---|---|
| Bare period | `/wiki/Mata_v._Avianca,_Inc.` |
| Percent-encoded | `/wiki/Mata_v._Avianca,_Inc%2E` |
| No trailing period | `/wiki/Regular_Article` |

**Key finding:** when clicking the percent-encoded link:

- **Firefox** displays `.` in the address bar (normalising for readability),
  but **copies `%2E` to the clipboard**. The decoded display is cosmetic only.
- **Chrome** displays `%2E` in the address bar as-is and copies `%2E` to the
  clipboard.

The fix holds up in both browsers. A URL generated by MediaWiki with `%2E`
will survive a copy-paste into a markdown context intact.

## Why not other punctuation?

Markdown parsers strip several other characters from the ends of auto-linked
URLs — notably `?`, `!`, `,` and `)`. Wikipedia article titles can end with
any of these (e.g.
[Do You Realize??](https://en.wikipedia.org/wiki/Do_You_Realize%3F%3F)).

These are already handled. `?` is not an unreserved character, so PHP's
`urlencode()` encodes it as `%3F`, and `wfUrlencode()` never decodes it back.
The same applies to `!`, `,`, and `)`. The trailing period is the odd one out:
`.` *is* unreserved, so `urlencode()` passes it through untouched and
`wfUrlencode()` has no reason to touch it either — making it the only common
title-ending punctuation that reaches the browser unencoded.

Parentheses deserve a special mention. While a `)` inside an explicitly written
markdown link like `[text](url_(with_parens))` can appear to break the syntax,
CommonMark-compliant parsers handle this elegantly: they count balanced
parentheses inside link destinations, so the `(with_parens)` pair is treated as
part of the URL and the final `)` correctly closes the markdown syntax. The
issue only affects older or non-compliant parsers. The trailing period has no
such saving grace — a bare `.` reads as sentence-ending punctuation to every
linkifier regardless of spec compliance.
